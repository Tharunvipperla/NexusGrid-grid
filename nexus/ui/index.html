    Building a NexusGrid-compatible relay
    
                
            
    
            
    
                
    
                    NexusGrid only cares that a relay does one job correctly: route framed messages between subscribed peers. You're free to add observability, rate-limit dashboards, online-status APIs, multi-region failover, whatever — as long as the base routing contract holds.
                
    
                
    Reference implementation: relay_server_modified.py in the NexusGrid repo. Read it end-to-end before forking — it's ~700 lines, every op is documented inline.
    
                
    Contract a NexusGrid-compatible relay must satisfy (Wave 40+):
    
                
      
                    
    1. WebSocket endpoint at /relay/<node_id> — clients connect with their node UUID baked into the URL.
      2. 
                    
    2. First frame is the register envelope: {"type":"register","grid_keys":[hex32,…],"display_name":"…",…}. Each entry in grid_keys is a 32-char lowercase hex digest derived from a context the client is a member of. An empty list is valid (fresh install).
      3. 
                    
    3. Maintain per-connection grid_key sets. Accept {"type":"add_grid_keys","grid_keys":[…]} and {"type":"remove_grid_keys","grid_keys":[…]} for live updates without reconnect.
      4. 
                    
    4. {"type":"relay","target":"<node_id>","payload":{…}} — point-to-point unicast. Forward to the target's connection as {"type":"relayed","from":"…","payload":…}.
      5. 
                    
    5. {"type":"broadcast","grid_key":"<hex32>","payload":{…}} — fan out only to other connected nodes whose grid_key set contains that key. Reject if the sender doesn't hold the key.
      6. 
                    
    6. Don't inspect, mutate, or persist payloads. Frames are end-to-end encrypted; the relay is a dumb router.
      7. 
                    
    7. {"type":"heartbeat"} → reply {"type":"heartbeat_ack"} so the RTT prober keeps your node in rotation.
      8. 
                    
    8. Optional: set a non-default NEXUS_GRID_KEY env to gate connection auth on a shared secret. Leave unset for an open relay anyone with a node_id can subscribe to.
      9. 
                
    
                
    Why per-context grid_keys (Wave 40)?
    
                
    
                    Until Wave 40, NexusGrid relayed everything under a single global grid_key. A rogue relay operator could see "node X is in groups A, B, C and pairs with Y, Z" because the metadata all rode the same bucket.
                
    
                
    
                    Wave 40 buckets every broadcast by grid_key, derived deterministically from the context's stable ID:
                
    
                
      
                    
    • grid_key_for_group(group_id) = sha256("nexus:group:" + group_id)[:32]
      • 
                    
    • grid_key_for_pair(a, b) = sha256("nexus:pair:" + sorted([a, b]).join("|"))[:32]
      • 
                
    
                
    
                    Two members of the same context independently produce the same key without coordinating. A relay operator now sees N independent buckets with no signal connecting them. End-to-end encryption inside each bucket remains the real security layer — the relay is just a dumb router that can't correlate.
                
    
                
    Operational expectations
    
                
      
                    
    • Reject grid_keys that aren't 32 lowercase hex chars (defensive — bad client / probe).
      • 
                    
    • Cap per-connection grid_key set at ≥ 256 (default in reference impl). Reject the op if exceeded.
      • 
                    
    • Cap individual frame size at ≥ 256 KB (reference default). Drop oversize payloads with an error frame.
      • 
                    
    • Rate-limit aggressively per-connection — both message count and broadcast count — to keep a noisy peer from starving everyone else. The reference impl uses a sliding 10s window.
      • 
                    
    • On WebSocket disconnect: clear the node from your registry AND drop any cached grid_keys. No background retry — clients reconnect on their own with fresh state.
      • 
                
    
                
    
                    Add extra ops (health, metrics, version) freely — NexusGrid ignores unknown type values. Just don't break the ones above.

                    A relay forwards group + pair traffic between peers that can't reach each other directly (NAT, firewalls, different networks). LAN-only peers don't strictly need one, but every group you intend to use across the internet does. You can change this any time from the Network → Local Relay panel.
                
    

                
    
                    Option A — Run a local relay on this node (recommended)
                    
    
                        Starts the in-process relay and opens a Cloudflare quick tunnel. Both bindings are usable immediately: ws://<your-LAN-ip>:port (LAN peers) and wss://…trycloudflare.com (cross-region peers). Each group later picks the lowest-RTT path automatically.
                    
    
                    
    
                        
                        
                        
                        Allowed 1024–65535. Default 9000 works for most users.
                    
    
                    
    
                        
                        
                    
    
                
    

                
    
                    Option B — Paste an existing relay URL
                    
    
                        Your friend's relay, a self-hosted one you built, or any community-run relay. It must start with ws:// or wss://.
